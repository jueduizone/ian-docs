# SPARK License 与 AI 使用可见性
## AI 时代开源可持续性框架

---

## 问题所在

传统开源 License 只管一件事：代码的使用权。它不管一个项目在大量依赖开源基础设施之后商业成功了会怎样。结果是一个有据可查的失败模式——商业上成功的项目极少回馈它赖以依赖的开源公共资源。

AI 编程工具带来了第二个结构性不同的问题。当开发者用 Cursor 或 Copilot 生成代码时，这些代码可能深度参考了某些开源项目的算法、模式和架构决策，却没有任何显式的 fork、import 或署名。依赖关系实际存在，只是不可见。

这是两个独立的问题，但有共同的根源：**我们用于追踪开源使用的工具，是为一个使用行为明确可追溯的世界设计的。那个世界已经不复存在。**

SPARK License 的设计目标就是同时解决这两个问题。

---

## SPARK License 是什么

SPARK License 是一种软件许可证，做了两件传统 License 做不到的事：

**1. 将经济义务与商业成功挂钩**

采用 SPARK License 意味着承诺将未来经济收益的一部分——来自营收、Token 发行或收购——回馈给项目所依赖的开源生态系统。这个义务不在采用时触发，而在商业化时触发。

这与现有方案有本质区别：
- GPL/Copyleft 约束的是代码衍生物，不是经济结果
- BSL 和 Commons Clause 限制商业使用，但不创造正向义务
- Open Source Pledge 是自愿且不可执行的

SPARK License 创造了一个前置承诺，链上记录，跟随项目走完整个商业生命周期。

**2. 将 AI 时代的使用定义为一类一级场景**

用 SPARK 许可项目训练 AI 模型、作为 AI 编程工具的检索来源、或作为 AI 生成代码的基础参考，均构成触发披露义务的使用场景。这填补了 GPL 从未被设计来覆盖的空白。

## AI 使用可见性问题

robots.txt 的类比有启发性，但并不完整。robots.txt 能起作用，是因为爬虫在 HTTP 请求这个明确的时间点访问内容，协议检查可以在这个时间点发生。AI 训练和推理没有这个时间点。当模型生成受某个开源项目影响的代码时，使用已经发生，分散在数百万条训练样本里，没有单一的归因节点。

这意味着**事后追责在结构上是不可能的**。现实的目标不是检测和惩罚，而是让使用变得足够可见，使规范得以形成，让合规成为阻力最小的路径。

四种机制，按近期可行性排序：

---

### 机制一：SPARK 元数据标准

每个 SPARK 许可项目在标准路径下包含一个机器可读的声明文件：

```
.well-known/spark.json
```

内容示例：
```json
{
  "project": "project-name",
  "registry": "0x...",
  "license": "SPARK-1.0",
  "ai_usage": "disclosure-required",
  "contact": "spark@project.org"
}
```

索引代码仓库的 AI 工具——Cursor、GitHub Copilot、Continue、Sourcegraph——本来就会爬取这类文件。当 AI 工具在代码生成过程中引用 SPARK 许可项目时，它可以：

1. 向开发者显示一条披露提示
2. 向 SPARK Registry 记录一条匿名化使用事件
3. 在生成的代码中附上署名注释

这不需要改变 License 的执行模型，只需要 AI 工具厂商实现一个轻量协议。激励结构很清晰：实现了协议的工具可以合理声称符合 SPARK 规范；不实现的工具则成为与许可证违规关联的那一方。

robots.txt 的类比在这里成立。robots.txt 在大多数司法管辖区没有法律约束力，但违反它是声誉和实际操作上的负担。SPARK 元数据的运作逻辑相同。

---

### 机制二：声明式 AI 依赖申报

对于使用 AI 编程工具的项目，SPARK License 要求每年提交一份 `AI_DEPENDENCIES.md` 声明——对 AI 生成代码中实质性参考了哪些开源项目的尽力估计。

这不是审计，而是自我证明，类似于碳足迹披露的运作方式。承认它并不精确。目的不是实现完美准确度，而是建立一种「主动审视」的习惯，以及一条让善意合规可见的记录。

SPARK 工具链提供一个扫描器：
- 分析 import 模式和代码结构
- 与 SPARK Registry 交叉比对
- 生成草稿 `AI_DEPENDENCIES.md` 供开发者审阅后提交

提交到 SPARK Registry 后，将更新 Dependency Funnel 中的依赖权重，将 1% 义务的一部分导向被识别为 AI 时代依赖的项目。

---

### 机制三：AI 工具协议集成

杠杆最大的干预点在 AI 工具层。面向 AI 编程工具的 SPARK 协议集成将：

- 在代码生成时：检查引用的代码是否源自 SPARK 许可仓库
- 向 SPARK Registry 记录匿名化使用事件（项目身份，不是用户身份）
- 为开发者提供一键披露流程

这类似于包管理器今天处理 License 合规的方式——npm 会在安装与项目不兼容的 License 包时发出警告。SPARK 协议将这一机制延伸到 AI 生成的代码。

目标集成方，按优先级排序：
1. Continue.dev（开源项目，集成最快）
2. Cursor（开发者心智份额最大，战略价值高）
3. GitHub Copilot（需要微软合作，长期目标）

数据模型在设计上保护隐私：Registry 记录某个 SPARK 许可项目在某次会话中被使用，不记录是谁在用或用在哪个项目里。使用频次用于影响 Dependency Funnel 权重，不识别或追踪任何个人开发者。

---

### 机制四：语义代码指纹

最长期的机制。在 SPARK 许可代码库中嵌入统计可检测的模式——不是可见标记，而是在算法设计、测试覆盖模式和命名约定中留下的结构性规律，能在 AI 工具的转化过程中存活下来。

当 AI 生成的代码中这些模式出现的频率超出基准线，就是该项目训练影响存在的证据（不是证明）。

这目前是研究阶段的能力。几个学术团队（MIT CSAIL、Stanford）已经验证了原理，但将其产品化为许可证执行系统的规模，是一个 3 至 5 年的时间跨度。

OpenSeed 将持续关注这一领域，并在研究成熟后将指纹能力纳入 SPARK 工具链。它不是当前实施方案的前置依赖。

## License 结构

SPARK License 1.0 分三个层级，由项目阶段触发：

**第 0 层——商业化前（默认）**

与 MIT 完全相同。无任何义务。完全自由地使用、修改、分发。

**第 1 层——商业活跃**

触发条件：年收入超过 $50,000，或 Token 发行（TGE）。

义务：
- 4% 经济收益导向 SPARK 对齐项目（通过 Registry）
- 1% 经济收益导向已申报的依赖项目
- 年度 SPARK 报告提交
- AI 使用披露（上述机制一和机制二）

**第 2 层——规模化（年收入 > $500 万）**

包含全部第 1 层义务，以及：
- 参与 SPARK 治理（多签或 DAO，视阶段而定）
- 强化依赖审计（不再自我申报，第三方核实）

**宽限期：** 商业化触发起 3 年。义务从第 1 天起累积，必须在第 3 年结束前全部结清。

**Token 项目：** 无论收入阶段，在 TGE 时托管 4%。

## 执行模型

SPARK License 的执行依靠三层机制，而非单一手段：

**第一层：链上声誉**

SPARK Registry 状态是公开的。处于 `Defaulted` 状态的项目无法获得 SPARK 资金池拨款，无法展示 SPARK 徽章，且可被公开搜索为不合规。对于依赖社区信任运作的项目——也就是绝大多数开源项目——这是真实的代价。

**第二层：生态压力**

SPARK 对齐的项目和资助方可以将采用该 License 作为资金、集成或合作的前提条件。这产生网络效应：随着 SPARK 覆盖面扩大，游离在外变成一种负担。

**第三层：法律可执行性（长期）**

SPARK License 中的商业义务条款在设计上具备法律可执行性，司法覆盖范围将随类似机制（BSL、Commons Clause）的判例法发展而扩大。OpenSeed 正与法律顾问合作，确保 License 文本从第一天起在关键司法管辖区（美国、欧盟、台湾、新加坡）可执行。

坦诚的评估：第三层在初期会比较弱。近期主要依靠第一层和第二层。这与 GPL 的发展轨迹相同——法律理论先于实际执行出现了好几年。机制在法院表态之前就能运作。

## 与 SPARK Pledge 的关系

SPARK License 和 SPARK Pledge 是进入同一生态系统的两个互补入口：

| | SPARK Pledge | SPARK License |
|--|--|--|
| 采用机制 | 自愿承诺 | 发布代码时附带 |
| 约束对象 | 采用它的项目 | 商业化使用许可代码的任何人 |
| 传播方式 | 受资助方也必须采用 | 随代码复用病毒式传播 |
| 执行手段 | 声誉 + 社会压力 | 声誉 + 法律 |
| AI 使用覆盖 | 否 | 是 |

两者共同创造两个增长向量：
- **Pledge：** 认同理念的项目主动加入生态系统
- **License：** 通过更广泛的软件供应链，依赖关系驱动的被动传播

项目可以同时采用两者。一个只使用了 SPARK 许可代码（但未签署 Pledge）的项目，在商业化时同样受 License 义务约束。

## 对 OpenSeed 路线图的意义

SPARK License 开发是第二阶段优先级，前提是 Registry 和 Funnel 基础设施已运行。

**第一阶段（当前 Grant 周期）：** SPARK Pledge 基础设施。建立 Registry、Dependency Funnel 和合规 Dashboard，用自愿采用者验证机制有效性。

**第二阶段（2026 年）：** SPARK License v1.0 发布。法律审查完成，AI 元数据标准发布，Continue.dev 集成上线。License 应用于所有 OpenSeed 开发的基础设施。

**第三阶段：** AI 工具的 SPARK 协议。Cursor 和/或 GitHub Copilot 集成。如可行，纳入语义指纹研究成果。

License 不需要等 AI 工具集成完成才有价值。每一个在 SPARK License 下发布的项目，都在依赖图中增加一个节点，在生态系统中增加一个未来义务。AI 可见性机制随时间推移提升这张图的质量。
