## 一、产品定位

**内部工具**，帮助内容创作者基于自己关注的方向，从多数据源中自动筛选内容、一键生成多平台可发布内容。

- 面向：内容运营、个人创作者
- 核心价值：从「有内容方向」到「各渠道可直接发布版本」全流程自动化
- 不是通用 AI 写作工具，是有数据源接入的垂直内容生产平台

---

## 二、用户体系

### 2.1 注册与激活

```
注册（邮箱+密码）→ 创建账号（可登录浏览）
→ 使用核心功能需要邀请码
→ 输入有效邀请码 → 激活完整权限
→ 首次登录引导弹窗：选择内容方向
→ 开始使用
```

**邀请码规则：**
- 管理员后台批量生成
- 每码限单人使用，一次性有效
- 有效期 30 天

### 2.2 用户 Profile

| 字段 | 说明 |
|------|------|
| email | 邮箱（唯一） |
| channels | 关注方向（主方向1个必选，副方向最多2个，共≤3个） |
| invite_code | 使用的邀请码 |
| activated_at | 激活时间 |

| RSS 订阅 | 用户自定义 | 已接入 |

### 4.2 用户自提交数据源

**流程：**
```
用户提交 URL（RSS / 网站）
→ AI 自动检测：能否爬取到内容
→ 进入人工审核队列（含 AI 检测结果作参考）
→ 审核通过/拒绝
→ Mailgun 邮件通知提交人
→ 审核通过 → 绑定到对应方向 → 加入数据源池
```

**规则：**
- 提交不限次数
- AI 判断结果仅作参考，不自动拒绝，全部进入人工审核
- 审核员可看到：URL、AI 爬取检测结果、内容样本预览

## 新增需求：管理员后台 - 用户使用统计

### 功能描述
管理员可在后台查看用户的 AI 使用记录，包括使用的模型、生成的内容、token 消耗等。

### 数据表设计

```sql
-- 使用记录表
CREATE TABLE IF NOT EXISTS usage_logs (
  id uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id uuid REFERENCES auth.users(id) NOT NULL,
  model text NOT NULL,                    -- 使用的 AI 模型
  platform text,                          -- 生成平台（小红书/公众号/Twitter/即刻）
  prompt_length integer,                  -- prompt 长度
  output_length integer,                  -- 输出长度
  tokens_used integer,                    -- 消耗的 token 数
  generation_time_ms integer,             -- 生成耗时（毫秒）
  status text DEFAULT 'success',          -- success / error / timeout
  error_message text,                     -- 错误信息（如果失败）
  created_at timestamptz DEFAULT now()
);

-- 索引
CREATE INDEX idx_usage_logs_user_id ON usage_logs(user_id);
CREATE INDEX idx_usage_logs_created_at ON usage_logs(created_at);
CREATE INDEX idx_usage_logs_model ON usage_logs(model);

-- 权限
GRANT ALL ON usage_logs TO anon, authenticated, service_role;
```

### 后台页面

**路径：** `/admin/usage`

**功能：**
- 查看所有用户的使用记录列表
- 按用户筛选
- 按模型筛选
- 按时间范围筛选
- 统计汇总（总 token 数、总生成次数、平均耗时）
- 导出 CSV

### 记录时机

在 `/api/generate` 调用 AI 生成内容时记录：
- 请求前记录：user_id、model、platform、prompt_length
- 请求后记录：output_length、tokens_used、generation_time_ms、status

### 统计指标

| 指标 | 说明 |
|------|------|
| 总生成次数 | 所有用户的生成总次数 |
| 总 token 消耗 | 累计消耗的 token 数 |
| 平均生成时间 | 单次生成的平均耗时 |
| 热门模型 | 使用频率最高的模型 |
| 活跃用户 | 生成次数最多的用户 |

## 方向切换功能设计（工作区模式）

### 核心概念
方向不是固定配置，而是可随时切换的"工作区"，类似 Notion 的 Workspace 切换。

### 设计方案

#### 1. 全局方向切换器
位置：NavBar 右侧，用户区域左侧

```
┌─────────────────────────────────────────────────────────┐
│ ⚡ ContentFlow  素材库 生成 工具  [🤖 AI ▼]  [用户头像] │
│                              方向切换器                  │
└─────────────────────────────────────────────────────────┘
```

点击后下拉选择：
- 🤖 AI
- 🌐 Web3
- 🔧 开源
- 🛠️ 开发者工具
- 📦 产品

#### 2. 切换后的变化

| 元素 | AI 方向 | 开源方向 |
|------|---------|---------|
| **素材库数据源** | BuildLog(ai) + HN + PH | HN + GitHub + 专项 RSS |
| **生成 Prompt** | "受众是对 AI 感兴趣的普通用户" | "受众是开发者社区" |
| **工具描述** | "AI 周报生成器" | "开源周报生成器" |
| **推荐内容** | AI 相关新闻 | 开源项目动态 |

#### 3. 数据隔离与共享

**隔离（按方向）**：
- 数据源过滤
- Prompt 上下文
- 推荐内容
- 工具文案

**共享（全局）**：
- 用户账号
- 历史记录
- 已生成内容
- 个人设置

#### 4. 技术实现

```typescript
// 方向状态管理
const [currentChannel, setCurrentChannel] = useState('ai')

// 切换方向
const switchChannel = (channel: string) => {
  setCurrentChannel(channel)
  localStorage.setItem('lastChannel', channel)
  // 刷新当前页面数据
  router.refresh()
}

// API 请求时带上方向
fetch(`/api/materials?channel=${currentChannel}`)
```

#### 5. 用户体验

- **记住选择**：localStorage 保存最后选择的方向
- **默认方向**：首次登录引导选择，后续默认用上次选择
- **快速切换**：无需刷新页面，数据实时更新
- **视觉反馈**：当前方向在 NavBar 高亮显示

## 小红书内容生成细化规格

### 核心原则

**小红书现在更推崇 400-600 字 + 强 hook，字数规格要根据内容类型区分。**

---

### 字数与结构模板（按内容类型）

| 内容类型 | 字数范围 | 结构模板 | Hook 示例 |
|---------|---------|---------|----------|
| **资讯类** | 400-550 字 | 事件 → 背景 → 影响 → 我的判断 → 行动建议 | "刚看到这个消息，太劲爆了" |
| **观点类** | 600-750 字 | 观点 → 论据×3 → 结论 | "说句实话，很多人对 AI 有误解" |
| **工具类** | 500-650 字 | 是什么 → 解决什么问题 → 怎么用 → 适合谁 → 我的评价 | "这个工具我用了 3 个月，真香" |

### Hook 要求

- **位置：** 正文第一句/第一段
- **作用：** 抓住用户注意力，制造好奇心或共鸣
- **类型：**
  - 数字型："看了 50 篇论文后，我发现了 3 个关键趋势"
  - 对比型："别人说 AI 会取代人，我不这么看"
  - 疑问型："为什么我劝你别急着买这个？"
  - 故事型："上个月我踩了个坑，分享给你避雷"
- **长度：** 1-2 句话，20-40 字

---

### 话题标签规格

| 标签类型 | 数量 | 选词规则 | 示例 |
|---------|------|---------|------|
| 大流量词 | 2 个 | 平台热搜/热门话题 | #AI #人工智能 |

## 热门内容发现系统 PRD（2026-04-09）

### 背景与问题

当前素材库有 3 种排序（时间/热度/增速），但存在以下问题：

| 问题 | 具体表现 |
|------|---------|
| **跨源数据无法横向比较** | HN score ≠ BuildLog hot_score，无法统一排序 |
| **score_delta 未实现** | 字段存在但没有计算逻辑，增速排序不准确 |
| **没有"正在爆"的信号** | 发现内容时已是热门，错过黄金创作窗口 |
| **多源出现未加权** | `_also_from` 已有字段但未计入排序 |

---

### 核心设计思路

**两个维度分开处理：**

- **热门内容**（绝对热度）→ 选题，话题量大，受众广
- **飙升内容**（相对增速）→ 蹭热点，时机好，竞争少

---

### 功能模块设计

#### 模块一：标准化热度分 `normalized_score`

**问题：** 不同数据源的分数量级不同，无法横向比较。

**解决方案：** 入库时将原始分映射为 0-100 的标准化分。

```typescript
function normalizeScore(source: string, rawScore: number): number {
  const ranges: Record<string, number> = {
    hackernews:  3000,   // HN 分数上限
    buildlog:    10000,  // BuildLog hot_score 上限
    producthunt: 500,    // PH votesCount 上限
    devto:       200,    // DEV.to reactions 上限
    arxiv:       0,      // 无互动数据
    huggingface: 0,      // 无互动数据
    openai:      0,      // 无互动数据
  }
  const max = ranges[source] ?? 100
  if (max === 0) return 50  // 无互动数据给基准分 50
  return Math.min(100, (rawScore / max) * 100)
}
```

**数据库变更：**
```sql
ALTER TABLE cf_materials ADD COLUMN normalized_score FLOAT DEFAULT 50;
```

**写入时机：** 在 `fetch-all` 的各个 `fetchXxx` 函数入库时计算写入。

---

#### 模块二：增速计算 `score_delta`（每小时 cron）

**问题：** 当前 `score_delta` 字段存在但没有被更新。

**解决方案：** 新增定时任务，每小时计算每条内容的分数增量。

**数据库变更：**
```sql
ALTER TABLE cf_materials ADD COLUMN score_at_1h_ago FLOAT;
ALTER TABLE cf_materials ADD COLUMN score_delta FLOAT DEFAULT 0;
```

**计算逻辑：**
```typescript
// 每小时执行（cron）
async function updateScoreDelta() {
  const { data } = await supabase
    .from('cf_materials')
    .select('id, score, score_at_1h_ago')
    .gte('published_at', new Date(Date.now() - 48 * 3600000).toISOString())
    .not('score', 'is', null)

  for (const item of data ?? []) {
    const delta = (item.score ?? 0) - (item.score_at_1h_ago ?? 0)
    await supabase.from('cf_materials').update({
      score_delta: delta,
      score_at_1h_ago: item.score,
    }).eq('id', item.id)
  }
}
```

**注意：** BuildLog 的 `hot_score` 本身包含增速预测，可以直接作为 `score_delta` 的参考映射：
```typescript
// BuildLog 数据入库时
score_delta: feed.hot_score ? normalizeScore('buildlog', feed.hot_score) : 0
```

#### 模块三：综合热度评分（排序权重）

借鉴 HN 算法，设计 ContentFlow 多信号综合评分：

```
综合热度分 =
  normalized_score × 0.5    // 互动热度：50%
  + freshness_score × 0.3   // 新鲜度：30%（越新越高）
  + cross_source_bonus × 0.2 // 跨源出现奖励：20%
```

**新鲜度计算：**
```typescript
// 时间衰减（参考 HN 算法）
function freshnessScore(publishedAt: string): number {
  const ageHours = (Date.now() - new Date(publishedAt).getTime()) / 3600000
  return Math.max(0, 100 - ageHours * 2)  // 每小时衰减 2 分，50 小时归零
}
```

**跨源奖励：**
```typescript
// _also_from 有值说明多个数据源都提到了这个内容
function crossSourceBonus(alsoFrom: string[]): number {
  const count = (alsoFrom?.length ?? 0) + 1
  return Math.min(100, count * 25)  // 每多一个源 +25 分，最多 100
}
```

#### 模块四：异常增速检测（Z-score）

**用途：** 发现分数突然暴涨的内容（"正在爆"的信号）。

**算法：**
```typescript
// 当某条内容的 score_delta 超出同批内容均值 2 个标准差，标记为"飙升"
function isViral(delta: number, avgDelta: number, stdDev: number): boolean {
  const zScore = (delta - avgDelta) / (stdDev || 1)
  return zScore > 2
}
```

**写入字段：**
```sql
ALTER TABLE cf_materials ADD COLUMN is_viral BOOLEAN DEFAULT false;
```

**触发时机：** 在增速 cron 执行后，对最近 48h 内容批量计算并更新 `is_viral`。
