---

## 一、产品定位

**内部工具**，基于 BuildLog 数据源，将信息流内容自动处理成可直接发布的多平台内容，减少人工排版和撰写时间。

- **不是通用 AI 写作工具**（市面已够多）
- **支持多渠道一键输出**：小红书、公众号、即刻、Twitter/X
- **核心价值：** 从「有内容线索」到「各渠道可直接发布版本」全流程自动化

---

## 二、数据来源

**BuildLog**（已有，不另建采集）

BuildLog `feeds` 表关键字段：
- `title` / `summary_zh` / `content_cleaned` — 内容本体
- `category`（build / ai）、`tags`、`ecosystem` — 分类
- `hot_score` / `twitter_likes` / `twitter_views` — 热度指标
- `developer_relevance` / `source_credibility` — AI 相关性分
- `published_at` — 时效性

选题评分：`hot_score + developer_relevance + 时效性衰减` 综合打分 → AI 推荐 Top 10 → 人工确认 1 题

---

## 三、核心流程

```
1. 选题
   BuildLog 数据（最近7天）
   → AI 评分 → Top 10 候选列表（含摘要 + 推荐理由）
   → 人工选定 1 题

2. 内容生成（一次生成，多渠道输出）
   → 小红书版：标题 + 正文 + 标签 + 竖版图
   → 公众号版：长文 + HTML 排版 + 封面图
   → 即刻版：短文 + 话题标签
   → Twitter/X 版：主推文 + 线程展开

3. 发布
   → 小红书/即刻：预览 + 一键复制（手动发）
   → 公众号：一键推送草稿到微信后台
   → Twitter/X：预览 + 一键复制（或接 API 自动发）
```

## 四、各渠道输出规格

### 4.1 小红书

**渠道特性：**
- 图文为王，封面图决定点击率
- 话题标签影响流量分发
- 用户偏好「干货+有用感」，不喜硬广

**输出规格：**
- 标题：≤20字，带 emoji，有钩子感（问句/数字/对比）
- 正文：300-500字，分点阐述，每点有结论
- 标签：10个（2-3个大流量词 + 5-6个垂直词 + 1-2个品牌词）
- 竖版图：1242 × 1660px，科技风（深色渐变背景 + 白字）
- 注意：禁止外链，不得有平台竞品名称

### 4.2 公众号

**渠道特性：**
- 以文章为主，深度阅读场景
- 标题影响打开率，标题党/数字型有效
- 搜索权重较高，SEO 价值大

**输出规格：**
- 标题：≤26字，含核心关键词
- 正文：1500-3000字，引言 + 3-5节正文 + 结语
- 封面图：900 × 500px
- 排版：wechat-md-editor 渲染（蓝色标题边框，简洁留白）
- 注意：不支持外链跳转，可用小程序码/二维码代替

### 4.3 即刻

**渠道特性：**
- 科技/创业垂直社区，用户质量高
- 字数限制约 1000 字，但主内容建议 300 字内
- 话题（Topic）是流量核心，要选对话题
- 评论互动率高，适合引发讨论

**输出规格：**
- 正文：200-400字，观点鲜明，有明确结论
- 话题：2-4个，选垂直度高的（AI工具、产品思考、开发者、创业）
- 图片：可附1-4张，建议附竖版图或截图
- 语气：自然口语化，像在和朋友分享想法
- 注意：不要写「干货」「必看」等营销味标题

### 4.4 Twitter/X

**渠道特性：**
- 字数限制 280 字（单条），可发线程（Thread）
- 前两行决定是否展开，必须有钩子
- 图片比纯文字互动率高约 150%
- 英文内容覆盖更广，中英双语可选

**输出规格：**
- 主推文：≤260字（留余量），带 1-2 个核心 hashtag
- Thread 展开：3-6条，每条自成一个观点
- 语言：优先英文，或中英双版本
- 图片：1张竖版图或信息图（1242×1660 或 1200×675）
- 注意：hashtag 不超过 3 个，过多降低互动率

### 4.5 SEO（博客/官网）

**用途：** OpenBuild 官网文章或博客发布

**输出规格：**
- 关键词：5-8个（主词 + 长尾词）
- meta description：≤160字
- SEO 评分：0-100（keyword density + 可读性 + 标题优化）
- 格式：Markdown，适配静态站点生成器

## 五、技术方案

### 5.1 架构
- **项目名：** FeedCast
- **独立项目**（数据直连 BuildLog Supabase 只读 key）
- Next.js + TypeScript
- 部署：Vercel（截图服务另部署）

### 5.2 可复用工具

| 工具 | 用途 |
|------|------|
| wechat-md-editor（`~/workspace/wechat-md-editor`）| Markdown → 公众号 HTML 排版 |
| seomachine（`~/workspace-work/seomachine`）| SEO 分析（keyword_analyzer、seo_quality_rater） |
| wechat-mp-publisher（OpenClaw Skill）| 一键推送公众号草稿 |
| social-copy-generator（OpenClaw Skill）| 多平台文案生成 |

### 5.3 图片生成
- 方案：**前端 `html-to-image` 库**，浏览器直接渲染 HTML 模板并导出 PNG
- 零服务器依赖，Vercel 直接部署，无需 Puppeteer 或 VPS
- 用户点「下载图片」直接在浏览器生成并保存
- 模板参数化：标题/正文/标签作为变量传入 HTML 模板

### 5.4 配置存储
- 所有敏感配置（AppID、AppSecret、API Key）写入 **Vercel 环境变量**
- 后台配置页：填入后写入环境变量，**不回显**（显示为 `••••••`）
- 代码中不硬编码任何 token/key
