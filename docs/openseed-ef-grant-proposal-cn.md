## 以太坊基金会资助申请

**申请方：** OpenSeed  
**联合创始人：** 林旅强、许银  
**申请金额：** $100,000 USD  
**资助类型：** 生态系统发展  
**执行周期：** 2025 年 5 月至 2025 年 11 月（6 个月）  
**部署目标：** Ethereum Mainnet

## 1. 执行摘要

SPARK Pledge 是 Vitalik Buterin 与 SPARK 工作组共同设计的递归式开源资助框架。目前它只是一份在私人 Signal 群中流传的草案，没有任何配套的基础设施，也没有项目正式加入。

OpenSeed 申请 $100,000，用于构建这套基础设施：链上 Pledge 注册合约、依赖资金漏斗、合规 Dashboard，以及一批来自华语开发者社区的种子项目。这笔资助将支持 SPARK 递归模型的首次真实验证。我们构建的基础设施将完全开源、无需许可、可被全球复用。

## 2. 问题

### 2.1 开源资助体系的困境

开源软件是以太坊生态每一层的基础：钱包、客户端、开发工具、库和协议。但支撑这些软件运转的资助体系长期未能跟上其重要性。

大多数资助是一次性的，没有延续性。商业化成功的项目很少将收益回流到它们所依赖的公共基础设施。关键组件由薪酬不足的维护者或志愿者支撑。现有机制各有边界：Gitcoin Grants 面向 Web3 原生项目和英语社区；RetroPGF 是 Optimism 生态内的回溯式资助；Protocol Guild 专注于以太坊核心协议贡献者。递归依赖层、非代币项目、非英语社区，这些空白至今没有人填。

### 2.2 SPARK 的设计

SPARK（Scaling Public Alignment with Recursive Kindling）是 Vitalik Buterin 与小型工作组设计的承诺框架。

加入 SPARK Pledge 的项目承诺将 4% 的未来经济收益用于资助其他 SPARK 对齐项目，递归传播：受资助方同样需要加入 Pledge。此外，项目承诺将 1% 用于资助其直接依赖的开源项目。3 年宽限期允许早期项目在具备商业可行性之前推迟履行义务，义务按收入阶段分级，而非一刀切强制执行。

Pledge 规范已在工作组内完成起草。目前缺失的，是注册项目、追踪合规、路由资金、解析依赖图所需的全套基础设施。

### 2.3 基础设施缺口

| 层级 | 现状 |
|------|------|
| SPARK Pledge 规范 | ✅ 草案已存在 |
| 链上 Pledge 注册合约 | ❌ 不存在 |
| 依赖资金漏斗工具 | ❌ 不存在 |
| 合规追踪与报告 | ❌ 不存在 |
| 已验证的种子项目 | ❌ 不存在 |
| 华语社区入口 | ❌ 不存在 |

OpenSeed 构建所有缺失的层。

### 2.4 AI 时代的特殊背景

AI 编程工具的普及使开源可持续性问题变得更加紧迫。METR 于 2025 年 7 月发布的随机对照研究显示，在复杂成熟代码库上工作的资深开发者，使用 AI 工具后任务完成时间反而增加了 19%。理解、验证和纠正 AI 输出所花费的成本，超过了生成本身带来的节省。

另一面是，AI 工具向开源仓库涌入了大量低质量 PR。到 2026 年初，curl、ghostty、tldraw 等知名项目已相继关闭或严格限制外部贡献。AI 降低了贡献代码的门槛，却没有降低审查代码的成本。维护者的角色正在从建设者转向守门人。

AI 让代码生产趋向商品化，真正稀缺的反而是另一些东西：长达数年的维护承诺、系统级的架构判断、社区建立的信任，以及判断什么代码值得存在的经验。SPARK 设计维系的正是这些。

现有开源评价指标，包括代码行数、提交数量、PR 量，在 AI 时代正在失去意义。OpenSeed 的依赖漏斗建立在 AI 无法伪造的信号上：项目存续年限、真实依赖深度和维护者连续性。

## 3. 我们在构建什么

### 3.1 SPARK Pledge 注册合约

部署在 Ethereum Mainnet 上的智能合约，使用 EAS（以太坊认证服务）将每个项目的 Pledge 采用记录为链上存证。合约存储动态 Pledge 参数，追踪商业化触发事件，管理宽限期计时器，并提供公开、无需许可的读取接口。

发行代币的项目：合约与 TGE 合约集成，在代币发行时自动托管 4% 的供应量，采用 Pledge 后无需任何手动操作。

非代币项目：Pledge 采用记录在链上，义务通过年度自我申报和向 SPARK 资金池转账的方式履行。

### 3.2 依赖资金漏斗

1% 的依赖分配需要解决两个问题：项目依赖哪些上游项目，以及如何对它们加权。

**依赖图解析**

漏斗集成 Google 的 deps.dev API，覆盖 npm、PyPI、Maven、Go 和 Cargo 五大生态，递归解析传递依赖，并通过 EAS 将 GitHub 仓库映射到链上身份。

**分配算法**

代码行数等传统指标在 AI 时代已不可靠。漏斗使用五个抗 AI 干扰的权重维度：

- 存续年限（25%）：项目持续活跃维护的年数。AI 可以一夜之间克隆一个库，但无法伪造五年的维护历史。
- 真实使用深度（30%）：实际依赖该包的下游项目数量，不是 Star，而是 dependents 数。数据来源：deps.dev。
- 维护者连续性（25%）：核心贡献者持续活跃的年限，衡量人的长期投入。
- Issue 响应质量（10%）：12 个月内的 issue 平均关闭时间和关闭率，维护者判断力的代理指标。
- 依赖深度衰减（10%）：直接依赖的权重高于间接依赖。

所有维度在加权前归一化到 0–1。算法完全开源，通过 OpenSeed 多签治理升级，每次变更均有链上记录。已注册 SPARK Registry 的项目获得 1.2 倍权重加成，激励 Pledge 在依赖图中传播。

**分配执行**

有链上身份的项目直接链上转账。无链上身份的项目，资金托管等待认领，12 个月无人认领后归还 SPARK 资金池。

### 3.3 合规 Dashboard

公开的 Web 界面，展示所有已注册 SPARK Pledge 项目的当前状态、商业化触发状态和宽限期倒计时、年度 SPARK 报告提交情况、资金流向记录，以及依赖漏斗分配明细。

任何利益相关方，包括资助方、社区成员和下游项目，都可以直接验证合规性，不需要信任 OpenSeed 作为中介。

### 3.4 SPARK 资金池

多签国库，接收本次 EF 资助和社区捐赠作为初始资金，接收商业化项目的 4% 递归贡献，通过审核流程向新的 SPARK Pledge 项目分配资金。OpenSeed 作为初期管理方运营，计划在第二阶段将治理去中心化。

### 3.5 华语社区试点（Cohort 1）

通过 OpenBuild 的开发者网络，招募并引导来自华语社区的 10 至 20 个开源项目加入。项目涵盖 Web3 工具、开发者基础设施、AI 工具和通用开源软件，不限于 Ethereum 原生项目。

每个入选项目将获得：Pledge 接入支持、技术集成协助、在 SPARK Registry 中的可见度，以及从 SPARK 资金池获得资助的资格。

**Cohort 1 筛选标准**

项目选择需要对以下三个问题给出肯定回答：

1. 项目是否有可识别的长期维护者，具备多年连续贡献历史？
2. 项目的核心价值是否建立在判断力和生态位上，而不是 AI 短期内可以复制的纯功能性代码？
3. 随着 AI 工具普及，项目的维护负担是否保持不变或有所增加？这是真实人工维护工作的直接证明。

这套标准筛选的是在 AI 饱和环境中价值不但得以存续、甚至变得更重要的项目。

## 4. 义务结构

在保护早期项目的同时确保最终贡献：

| 年收入阶段 | SPARK 义务 |
|-----------|-----------|
| 低于 $50,000 | 不触发 |
| $50K 至 $500K | 2%（早期阶段折扣率） |
| $500K 至 $5M | 4%（标准费率） |
| 超过 $5M | 4% 加治理参与义务 |

代币项目：无论收入阶段，在 TGE 时托管 4%。代币代表未来的经济收益，而非当期收入。

宽限期：从商业化触发起 3 年。义务从第 1 天开始累积，必须在第 3 年结束前全部结清。

## 5. 为什么是 OpenSeed

### 5.1 社区根基

OpenSeed 由 OpenBuild、开源社（中国最大的厂商中立开源社区）、PyChina 和 RustCC 的团队共同创立。这些组织在中国大陆、台湾、香港及更广泛的华语开发者群体中合计代表数十万开发者。

OpenBuild 已通过开发者教育、黑客松和项目孵化将数千名开发者引入以太坊生态。这一现有的信任网络是第一阶段 SPARK 推广的核心分发渠道。

### 5.2 直接参与 SPARK 设计

OpenSeed 团队自 2025 年 4 月起加入 Vitalik Buterin 发起的 Signal 群，直接参与了形成当前 SPARK Pledge 草案的机制设计讨论。我们不是在解读一份二手文件，而是在执行我们自己参与设计的方案。

### 5.3 与 SPARK 工作组的关系

我们将自身定位为三个角色：SPARK Pledge 基础设施的首个和主要实施方；向工作组提供数据反馈的闭环，我们的试点数据将直接为机制完善提供依据；以及管理者而非所有者。SPARK Pledge 规范是开放的，任何组织都可以实施，我们在构建参考实现。

### 5.4 与现有机制的互补性

| 机制 | 资金来源 | 递归性 | 覆盖范围 |
|------|---------|--------|---------|
| Gitcoin QF | 社区捐赠 | 否 | Web3，英语为主 |
| Optimism RetroPGF | OP 代币增发 | 否 | Optimism 生态 |
| Protocol Guild | 项目捐赠 | 否 | 以太坊核心开发者 |
| Drips Network | 任意捐赠方 | 部分 | 任意，手动配置 |
| Open Source Pledge | 企业预算 | 否 | 任意，自愿 |
| SPARK / OpenSeed | 收益比例承诺 | 是 | 任意开源项目，全球 |

OpenSeed 填补的是上述所有机制都未覆盖的那个层。

## 8. 长期愿景

本次资助用于基础设施建设和试点验证。自我持续的路径在于递归机制本身：随着 Cohort 1 项目陆续商业化，4% 回流至 SPARK 资金池。20 个 Cohort 1 项目中，哪怕只有 3 个达到商业规模并履行 Pledge，资金池就能实现自我补充，无需继续依赖外部资助。这是对机制有效性的最终验证。

OpenSeed 通过可选的 Registry 增值服务、生态合作和资助支持获取运营收入。链上基础设施无需许可，即便 OpenSeed 作为组织不复存在，它依然可以继续运行。

第二阶段（2026 年）：扩大规模至全球 100 个以上项目；作为互补合规层与 Gitcoin Grants 和 RetroPGF 集成；将 SPARK 资金池治理过渡至链上 DAO；发布供其他生态复用的开源工具。

本次资助构建的所有基础设施以 MIT 许可证开源：智能合约在 GitHub 和 Etherscan 上公开验证，依赖漏斗算法完全开放可审计，Dashboard 前端开源，SPARK Pledge 工具可被任何生态 fork 和部署。
