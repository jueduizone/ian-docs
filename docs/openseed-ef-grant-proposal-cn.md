# OpenSeed：构建 SPARK Pledge 的实施基础设施
## 以太坊基金会资助申请

**申请方：** OpenSeed  
**联合创始人：** 林旅强、许银  
**申请金额：** $100,000 USD  
**资助类型：** 生态系统发展  
**执行周期：** 2025 年 5 月 – 2025 年 11 月（6 个月）  
**部署目标：** Ethereum Mainnet

## 1. 执行摘要

SPARK Pledge 是由 Vitalik Buterin 与 SPARK 工作组共同设计的递归式开源资助框架。目前它只是一份在私人 Signal 群中流传的草案文件，没有任何实施基础设施，没有任何项目正式加入。

OpenSeed 申请 $100,000，用于构建这套基础设施：链上 Pledge 注册合约、依赖资金漏斗、合规 Dashboard，以及一批来自华语开发者社区的种子项目。这笔资助将支持 SPARK 递归模型的第一次真实验证。我们构建的基础设施将完全开源、无需许可、可被全球复用。

## 2. 问题

### 2.1 开源资助体系已经失效

开源软件是以太坊生态每一层的基础：钱包、客户端、开发工具、库和协议。但支撑这些软件的资助体系没有跟上：

大多数资助是一次性的，没有延续性。商业化成功的项目很少把收益回流到它们所依赖的公共资源。关键基础设施由薪酬不足或志愿者维护者支撑。现有机制——Gitcoin Grants、RetroPGF、Protocol Guild——各自解决了部分问题，但留下了显著的空白，尤其是非代币项目、非英语社区和递归依赖层。

### 2.2 SPARK 的提案

SPARK（Scaling Public Alignment with Recursive Kindling）是由 Vitalik Buterin 与小型工作组设计的承诺框架，核心机制如下：

加入 SPARK Pledge 的项目承诺将 4% 的未来经济收益用于资助其他 SPARK 对齐项目，该机制具有递归性：受资助方也必须加入 Pledge。项目还额外承诺将 1% 的收益用于资助其直接依赖的开源项目。3 年宽限期允许早期项目推迟履行义务，直至商业化可行。义务按收入阶段分级，而非一刀切。

Pledge 规范已在工作组内完成起草和审阅。目前不存在的是：注册项目、追踪合规、路由资金、解析依赖图所需的任何基础设施。

### 2.3 基础设施缺口

| 层级 | 现状 |
|------|------|
| SPARK Pledge 规范 | ✅ 草案已存在 |
| 链上 Pledge 注册合约 | ❌ 不存在 |
| 依赖资金漏斗工具 | ❌ 不存在 |
| 合规追踪与报告 | ❌ 不存在 |
| 已验证的种子项目 | ❌ 不存在 |
| 华语社区入口 | ❌ 不存在 |

OpenSeed 构建所有缺失的层。

### 2.4 为什么现在更重要

AI 编程工具的兴起带来了结构性转变，使开源可持续性问题更加紧迫。METR 于 2025 年 7 月发布的一项随机对照研究发现，对于在复杂、成熟代码库上工作的资深开发者，AI 工具使任务完成时间增加了 19%。理解、验证和纠正 AI 输出的成本超过了生成带来的节省。

与此同时，AI 工具向开源仓库涌入了大量低质量 PR。到 2026 年初，curl、ghostty、tldraw 等项目已关闭或严格限制外部贡献。AI 降低了贡献代码的门槛，但没有降低审查代码的成本。维护者正从建设者转变为守门人。

这改变了开源价值的含义。代码生产越来越被商品化。稀缺的是长期维护承诺、架构判断力、社区信任，以及判断什么代码应该存在的人类智识。这些正是 SPARK 设计用来维系的东西。

这也改变了开源价值的衡量方式。现有指标——代码行数、提交数量、PR 量——正在成为噪音。OpenSeed 的依赖漏斗建立在 AI 无法伪造的信号上：项目存续年限、真实依赖深度和维护者连续性。

## 3. 我们在构建什么

### 3.1 SPARK Pledge 注册合约

部署在 Ethereum Mainnet 上的智能合约，使用 EAS（以太坊认证服务）将每个项目的 Pledge 采用记录为链上存证。注册合约存储动态 Pledge 参数、追踪商业化触发事件、管理宽限期计时器，并提供公开、无需许可的读取接口。

对于发行代币的项目，注册合约与 TGE 合约集成，在发行时自动托管 4% 的代币供应量，采用 Pledge 后无需任何手动操作。

对于非代币项目，Pledge 采用记录在链上，义务通过年度自我申报和向 SPARK 资金池转账的方式履行。

### 3.2 依赖资金漏斗

1% 的依赖分配需要知道一个项目依赖哪些上游项目以及如何对它们加权。

**依赖图解析**

漏斗集成 Google 的 deps.dev API，覆盖 npm、PyPI、Maven、Go 和 Cargo 五大生态。递归解析传递依赖，并通过 EAS 将 GitHub 仓库映射到链上身份。

**分配算法**

代码行数等传统指标已不再是开源价值的可靠信号。漏斗使用五个抗 AI 干扰的权重维度：

- 存续年限（25%）：项目持续活跃维护的年数。AI 可以一夜之间克隆一个库，但无法伪造五年的维护历史。
- 真实使用深度（30%）：实际依赖该包的下游项目数量，不是 Star 数，而是 dependents 数。数据来源：deps.dev。
- 维护者连续性（25%）：核心贡献者持续活跃的年限，衡量人的长期投入。
- Issue 响应质量（10%）：12 个月内的 issue 平均关闭时间和关闭率，维护者判断力的代理指标。
- 依赖深度衰减（10%）：直接依赖的权重高于间接依赖。

所有维度在加权前归一化到 0–1。算法完全开源，通过 OpenSeed 多签治理升级，每次变更均有链上记录。已注册 SPARK Registry 的项目获得 1.2 倍权重加成，激励 Pledge 在依赖图中传播。

**分配执行**

有链上身份的项目：直接链上转账。无链上身份的项目：资金托管至认领，或在 12 个月无人认领后归还 SPARK 资金池。

### 3.3 合规 Dashboard

公开的 Web 界面，展示所有已注册 SPARK Pledge 项目及其当前状态、商业化触发状态和宽限期倒计时、年度 SPARK 报告提交情况、资金流向透明记录以及依赖漏斗分配记录。

任何利益相关方——资助方、社区成员、下游项目——都可以在不依赖 OpenSeed 作为中介的情况下验证合规性。

### 3.4 SPARK 资金池

多签国库，接收本次 EF 资助和社区捐赠作为初始资金，接收商业化项目的 4% 递归贡献，通过审核流程向新的 SPARK Pledge 项目分配资金。OpenSeed 作为初期管理方运营，计划在第二阶段将治理去中心化。

### 3.5 华语社区试点（Cohort 1）

通过 OpenBuild 的开发者网络，招募并引导来自华语社区的 10–20 个开源项目。项目涵盖 Web3 工具、开发者基础设施、AI 工具和通用开源软件，不限于 Ethereum 原生项目。

每个项目获得：Pledge 接入支持、技术集成协助、在 SPARK Registry 中的可见度，以及从 SPARK 资金池获得资助的资格。

**Cohort 1 筛选标准**

项目选择需要对以下三个问题给出肯定回答：

1. 项目是否有可识别的长期维护者，具备多年贡献历史？
2. 项目的核心价值是否根植于判断力和生态位，而非 AI 数月内可复制的纯功能性代码？
3. 随着 AI 工具的普及，项目的维护负担是否保持不变或有所增加——这是真实人工维护工作的证明？

这些标准筛选的是在 AI 饱和环境中价值得以存续、甚至变得更重要的项目。

## 4. 义务结构

在保护早期项目的同时确保最终贡献：

| 年收入阶段 | SPARK 义务 |
|-----------|-----------|
| 低于 $50,000 | 不触发 |
| $50K 至 $500K | 2%（早期阶段折扣率） |
| $500K 至 $5M | 4%（标准费率） |
| 超过 $5M | 4% 加治理参与义务 |

代币项目：无论收入阶段如何，在 TGE 时托管 4%。代币代表未来的经济收益，而非当期收入。

宽限期：从商业化触发起 3 年。义务从第 1 天开始累积，必须在第 3 年结束前全部结清。

## 5. 为什么是 OpenSeed

### 5.1 社区根基

OpenSeed 由 OpenBuild、开源社（中国最大的厂商中立开源社区）、PyChina 和 RustCC 的团队共同创立。这些组织在中国大陆、台湾、香港及更广泛的华语开发者社区中合计代表数十万开发者。

OpenBuild 已通过开发者教育、黑客松和项目孵化将数千名开发者引入以太坊生态。这一现有的信任网络是第一阶段 SPARK 推广的分发渠道。

### 5.2 直接参与 SPARK 设计

OpenSeed 团队自 2025 年 4 月起加入 Vitalik Buterin 发起的 Signal 群，直接参与了形成当前 SPARK Pledge 草案的机制设计讨论。我们构建的是基于一手设计知识的实施，而非对二手解读的执行。

### 5.3 与 SPARK 工作组的关系

我们将自身定位为三个角色：SPARK Pledge 基础设施的首个和主要实施方；向工作组提供反馈的回路，我们的试点数据将为机制完善提供依据；管理者而非所有者。SPARK Pledge 规范是开放的，任何组织都可以实施它，我们在构建参考实现。

### 5.4 与现有机制的互补性

| 机制 | 资金来源 | 递归性 | 覆盖范围 |
|------|---------|--------|---------|
| Gitcoin QF | 社区捐赠 | 否 | Web3，英语为主 |
| Optimism RetroPGF | OP 代币增发 | 否 | Optimism 生态 |
| Protocol Guild | 项目捐赠 | 否 | 以太坊核心开发者 |
| Drips Network | 任意捐赠方 | 部分 | 任意，手动配置 |
| Open Source Pledge | 企业预算 | 否 | 任意，自愿 |
| SPARK / OpenSeed | 收益比例承诺 | 是 | 任意开源项目，全球 |

OpenSeed 是上述任何机制都未提供的那个实施层。

## 8. 长期愿景

本次资助支持基础设施构建和试点验证。自我持续的路径在于递归机制本身：随着 Cohort 1 项目商业化，4% 回流至 SPARK 资金池。如果 20 个 Cohort 1 项目中哪怕有 3 个达到商业规模并履行 Pledge，资金池将无需进一步资助即可自我补充。这是对机制的终极验证。

OpenSeed 通过可选的 Registry 增值服务、生态合作和资助支持获取运营收入。链上基础设施无需许可，可以在 OpenSeed 作为组织消亡后继续运行。

在第二阶段（2026 年），我们将把规模扩大至全球 100 个以上项目，作为互补合规层与 Gitcoin Grants 和 RetroPGF 集成，将 SPARK 资金池治理过渡至链上 DAO，并发布供其他生态复用和部署自有 SPARK 实施的开源工具。

本次资助构建的所有基础设施将以 MIT 许可证开源：智能合约在 GitHub 和 Etherscan 上验证公开，依赖漏斗算法完全开放可审计，Dashboard 前端开源，SPARK Pledge 工具可被任何生态 fork 和部署。
