# OpenSeed：将 SPARK Pledge 从理念落地为基础设施

**联合创始人：** 林旅强、许银、辛庆  
**部署目标：** Ethereum Mainnet

---

## 1. 问题

### 1.1 开源的钱从哪里来

以太坊生态每一层都建在开源软件上：钱包、客户端、开发框架、底层库。没有这些，就没有现在的以太坊。但这些软件的资金来源，长期处于一种尴尬状态。

资助大多是项目制的，给完就结束。靠开源起家、后来商业化的项目，几乎没有将收益回馈给上游的动力或机制。关键基础设施的维护者，要么拿着远低于市场的薪酬，要么干脆是无偿的志愿者。

现有的几个机制各有局限：Gitcoin Grants 依赖社区捐赠；Optimism RetroPGF 是生态内的回溯式资助；Protocol Guild 只针对以太坊核心协议的贡献者。递归依赖层、非代币项目、非英语社区，这些区域至今是空白。

### 1.2 SPARK 的思路

SPARK（Scaling Public Alignment with Recursive Kindling）的核心是一个承诺：加入的项目，将 4% 的未来经济收益用于资助其他 SPARK 对齐项目，收到资助的项目同样承诺同样的比例，递归传播。同时，项目再承诺 1%，专门流向它自己直接依赖的开源项目，让供养自己的上游也能拿到钱。

为了不把早期项目压垮，设置了 3 年宽限期，义务按收入阶段分级触发而不是立刻生效。

规范文本已经写好了，工作组内部也完成了讨论。缺的是让这套机制真正运转起来的一切：没有注册入口，没有链上存证，没有依赖图解析，没有合规追踪，没有第一批愿意加入的项目。

### 1.3 缺口在哪里

| 层级 | 现状 |
|------|------|
| SPARK Pledge 规范 | ✅ 草案已完成 |
| 链上 Pledge 注册合约 | ❌ 不存在 |
| 依赖资金漏斗工具 | ❌ 不存在 |
| 合规追踪与报告 | ❌ 不存在 |
| 已验证的种子项目 | ❌ 不存在 |
| 华语社区入口 | ❌ 不存在 |

OpenSeed 来填这些空白。

### 1.4 AI 时代让这件事更迫切

过去几年，AI 编程工具的普及在开源社区引发了一个意料之外的副作用。METR 在 2025 年 7 月的随机对照研究发现，对于在复杂成熟代码库上工作的资深开发者，使用 AI 工具之后任务完成时间反而增加了 19%。AI 生成的代码需要有人理解、核验、纠错，这个成本比生成本身更高。

与此同时，低质量 AI 生成的 PR 开始批量涌入各个开源仓库。curl、ghostty、tldraw 等项目到 2026 年初已相继收紧甚至关闭外部贡献通道。AI 让"提交代码"变得更容易，却让"审查代码"变得更累。

代码越来越容易生成，但坚持数年的维护、判断一个系统该往哪里走的能力、长期积累的社区信任，这些没有变。SPARK 保护和激励的，正是这部分。

传统的开源评价指标，比如代码行数、提交频率、PR 数量，在 AI 时代正在失效。OpenSeed 的依赖漏斗绕开这些，专注于 AI 伪造不了的信号：项目存续多少年、有多少真实下游依赖、核心维护者连续活跃了多久。

## 2. 我们在构建什么

### 2.1 SPARK Pledge 注册合约

部署在 Ethereum Mainnet 上的智能合约，将每个项目的 Pledge 采用通过 EAS（以太坊认证服务）记录为链上存证。合约负责存储各项目的 Pledge 参数，追踪商业化触发事件，管理宽限期计时，并提供公开、无需许可的查询接口。

发行代币的项目，合约与 TGE 合约直接集成：代币发行时自动托管 4% 供应量，签署 Pledge 之后不需要任何手动操作。非代币项目，Pledge 记录在链上，义务通过年度自我申报和向资金池转账的方式履行。

### 2.2 依赖资金漏斗

1% 的依赖分配要解决两个问题：这个项目依赖了哪些上游，以及这些上游各自该拿多少。

**依赖图解析**

漏斗接入 Google 的 deps.dev API，覆盖 npm、PyPI、Maven、Go、Cargo 五大生态，递归展开传递依赖，并通过 EAS 将 GitHub 仓库身份映射到链上地址。

**分配算法**

代码行数等旧指标在 AI 时代已经失真，漏斗改用五个维度，每一个都是 AI 短期内伪造不了的：

- 存续年限（25%）：项目持续活跃的年数。AI 可以一夜克隆出一个库，伪造不了五年的维护历史。
- 真实依赖深度（30%）：实际将该包列为依赖的下游项目数量，不是 Star，是真实使用数据。来源：deps.dev。
- 维护者连续性（25%）：核心贡献者持续活跃的年限，衡量的是人对项目的长期投入。
- Issue 响应质量（10%）：12 个月内的 issue 平均关闭时间和关闭率，是维护者判断力和责任感的代理指标。
- 依赖深度衰减（10%）：直接依赖的权重高于间接依赖，越近的上游影响越大。

所有维度加权前归一化到 0–1。算法完全开源，通过 OpenSeed 多签治理升级，每次变更都有链上记录可查。已加入 SPARK Registry 的项目获得 1.2 倍权重加成，激励 Pledge 在依赖链中扩散。

**资金分发**

有链上身份的项目直接转账；没有链上身份的，资金在多签地址里等待认领，超过 12 个月无人认领则归还 SPARK 资金池。

### 2.3 合规 Dashboard

一个公开的 Web 页面。注册项目的合规状态、宽限期倒计时、SPARK 报告记录、资金流向、漏斗分配结果——全部在这里，实时可查。

任何人，社区成员、下游项目、合作机构，都可以直接在这里核查合规情况，不需要信任 OpenSeed 的任何表态。

### 2.4 SPARK 资金池

多签国库，接收社区捐赠，后续接收商业化项目的 4% 递归贡献，通过审核流程向新加入的 SPARK 项目分配资金。OpenSeed 作为初期管理方，第二阶段逐步过渡到链上治理。

### 2.5 华语社区试点（Cohort 1）

通过 OpenBuild 的开发者网络，首期招募 10 至 20 个华语开源项目。范围不限于 Ethereum 原生项目，Web3 工具、开发者基础设施、AI 工具、通用开源软件都在考虑范围内。

入选项目将获得：Pledge 接入支持、技术集成协助、在 SPARK Registry 中的曝光，以及申请 SPARK 资金池资助的资格。

**Cohort 1 筛选标准**

以下三个问题，需要都能回答"是"：

1. 项目有没有可识别的长期维护者，有多年连续贡献记录的那种？
2. 项目的核心价值建立在判断力和生态位上，还是纯粹是 AI 几个月内就能复制的功能代码？
3. AI 工具普及之后，项目的维护负担有没有增加或保持不变？这是真实人工投入的直接证明。

选进来的项目，是那些在 AI 普及的环境下价值不降反升的项目。

### 2.6 SPARK License：义务随代码传播

Pledge 要求项目主动找到入口、理解机制、完成签署。这条路对理念认同的项目有效，对其他人摩擦太大——不是不认同，只是永远不会迈出那一步。

SPARK License 是第二条入口，设计逻辑是被动传播而不是主动加入。

传统开源许可证只管代码的使用权：GPL 管衍生代码，BSL 限制商业使用，Open Source Pledge 是道义呼吁。没有哪个将经济义务附在项目的商业化生命周期上。SPARK License 把义务结构直接嵌进许可证文本：任何包含了 SPARK License 代码的下游项目，商业化后自动继承 4% 递归义务和 1% 依赖义务。用了代码，就带着这个承诺，不需要额外的主动行动。

商业化之前，SPARK License 等同于 MIT：零义务，完全自由使用、修改、分发。触发条件和 Pledge 路径一致，分级义务结构也相同。

SPARK License 还明确覆盖了 GPL 从未设计处理的场景：AI 时代的使用方式。将 SPARK License 项目作为训练数据源、AI 编程工具的检索参考、或 AI 生成代码的架构模式基础，都构成触发申报义务的使用场景。这是第一个将 AI 中介视为一等公民依赖类别的许可证。

**Pledge 和 License 是独立的两条入口，不是捆绑关系。** 签了 Pledge 的项目不需要换许可证，用 SPARK License 发代码的项目也不需要单独签 Pledge。两条路设计为互补叠加，同时采用的项目获得漏斗 1.3 倍权重加成——这是经济激励，不是强制要求。

**OSI 认证路径**

SPARK License 的长期目标是进入开源许可证的标准体系，申请 OSI 认证和 SPDX 标识符。法律论据是明确的：SPARK License 满足开源定义（OSD）全部十条，经济义务仅在商业化后触发，精神上更接近 copyleft 而非商业限制。

### 2.7 SPARK.md：修复 AI 时代的依赖图盲区

漏斗的 1% 依赖分配有一个前提假设：能看到依赖是谁。过去这个假设成立——引入一个包，它出现在 manifest 里，依赖图是可追溯的。

AI 编程工具打破了这个假设。开发者用 Cursor 或 Copilot 生成代码，生成结果可能深度借鉴了某个开源项目的算法、设计模式或架构决策，但不会产生任何 import 语句，也不会出现在任何 manifest 里。依赖关系在实质意义上存在，但对所有现有工具都不可见。

这不是边缘情况，这是 2025 年 AI 辅助开发的默认状态。模式被 AI 反复复用的项目，可能在漏斗里拿到最少，因为它们的影响力是隐形的。

SPARK.md 是让这种影响力变得可见的机制。每个 SPARK 注册仓库的根目录放一个 `SPARK.md` 文件。YAML frontmatter 供扫描器和工具链机器解析；Markdown 正文是给人和 AI 编程助手的自然语言上下文——Cursor、Copilot、Continue 等工具索引仓库时会加载这部分内容。

AI 依赖申报是尽力估计，不是完整审计。开发者看一下生成的代码，判断哪些上游项目可能对它有影响，记录下来。提交本身就视为合规。目标不是精确，是建立一种习惯：像对待包依赖一样对待 AI 依赖。

SPARK.md 设计为独立的开放标准传播，类比 `security.txt`（RFC 9116）：放文件是单方面的动作，声明意愿，提供上下文，覆盖越广，漏斗的信号就越准确。

### 2.8 义务结构

| 年收入 | SPARK 义务 |
|--------|-----------|
| 低于 $50,000 | 不触发 |
| $50K 至 $500K | 2%（早期折扣率） |
| $500K 至 $5M | 4%（标准费率） |
| 超过 $5M | 4% 加治理参与义务 |

代币项目：TGE 时托管 4%，不看收入阶段。代币代表的是未来收益，不是当期收入。

宽限期：商业化触发起算 3 年，义务从第 1 天开始累积，第 3 年结束前必须全部结清。

## 3. 与现有机制的关系

| 机制 | 资金来源 | 承诺方式 | 递归性 | 全球开放 |
|------|---------|---------|--------|---------|
| Gitcoin QF | 社区捐赠 | 无前置承诺 | 否 | 部分 |
| Optimism RetroPGF | OP 代币增发 | 无前置承诺 | 否 | 否 |
| Protocol Guild | 项目捐赠 | 无前置承诺 | 否 | 否 |
| Drips Network | 任意捐赠方 | 手动配置 | 部分 | 是 |
| Open Source Pledge | 企业预算 | 道义呼吁 | 否 | 是 |
| **SPARK / OpenSeed** | **收益比例承诺** | **前置 + 链上存证** | **是** | **是** |

这几个机制各自解决了一部分问题，但没有一个同时满足：前置承诺、递归传播、链上可验证、面向全球任意开源项目。OpenSeed 是补位，不是竞争。

## 4. 我们不做什么

边界同样重要。

- **不造新链或 L2。** 部署在 Ethereum Mainnet 上。
- **不审计项目收入。** 机制依靠自我申报和社区问责运转。对于非代币项目，社会和声誉压力是执行层，不是监控。
- **不取代 Gitcoin、RetroPGF 或 Protocol Guild。** 我们提供的是它们都不提供的那一层：前置承诺、递归传播、全球开放。设计上是互补，不是竞争。
- **不限于 Web3 项目。** 任何有真实使用量和真实维护者的开源项目都可以加入。
- **不声称解决了所有开源资金问题。** SPARK 是一个机制，解决了问题的一部分。我们不是最终答案。

## 5. 开源承诺

OpenSeed 构建的所有基础设施以 MIT 协议开源。合约在 GitHub 和 Etherscan 公开验证，漏斗算法可审计，工具链可以被任意生态 fork 去跑自己的 SPARK 实施。

这不是加分项，是底线。为开源构建的基础设施本身必须开源，没有商量余地。

## 6. 路线图

**Phase 1：基础（2026 年 5–6 月）**
完成合约架构设计，部署 Registry v0.1 至测试网，上线 Pledge 接入网站，启动 Cohort 1 招募。同步起草 SPARK License 法律文本，发布 SPARK.md 规范 v0.1。

**Phase 2：核心上线（2026 年 7–8 月）**
Registry 合约上 Mainnet，依赖漏斗 v0.1 上线（先覆盖 npm 和 PyPI），合规 Dashboard v0.1 上线，Cohort 1 完成 10 个以上项目接入、Pledge 签署和 SPARK.md 部署。SPARK License v0.9 分发给 Cohort 1 项目。

**Phase 3：试点验证（2026 年 9–11 月）**
Cohort 1 完成首轮 SPARK 报告，依赖漏斗处理首批分配（含 AI 依赖信号），面向全球开放注册。

**Phase 4：规模化**
规模扩展到全球 100 个以上项目，与 Gitcoin 和 RetroPGF 作为合规互补层集成，资金池治理过渡至链上 DAO，开放工具链供其他生态 fork 和部署自己的 SPARK 实施。
