pros-cons

Hermes 的优缺点，一页讲清

优点

学习闭环：记忆和 skills 会让系统越用越顺。
多入口：CLI、IM、API、cron 适合不同场景。
可拆角色：profile 能承载不同职责和工具权限。
模型分层：主模型、辅助模型、embedding 可分别优化。
可运营：能做健康检查、配置审计、故障恢复。

缺点

配置复杂：provider、gateway、memory、systemd 互相影响。
故障面多：模型、代理、数据库、IM、session 都可能出问题。
A2A 不天然完美：不同入口的 session 不自动统一。
需要纪律：不验证、不沉淀、不进任务系统，就退化成聊天 bot。
旧事实污染：长期记忆需要版本化和清理。

recommended-workflows

我们现在最推荐的工具 / 工作流组合

这些不是 Hermes 独有概念，但和 Hermes 的工具、记忆、skills、gateway 组合起来，最容易产生稳定收益。

workflow 01
Hindsight + Memory 分层
短偏好进常驻 memory；长事实和历史决策进 Hindsight；流程方法进 skills。适合所有长期使用者。
workflow 02
Skills as SOP
把排障、部署、PPT、代码审查、发布流程写成 skill。团队经验不再散落在聊天记录里。
workflow 03
Multica / Issue 驱动
跨人/跨 agent 的任务进入 issue；状态、owner、验收标准都落地，而不是只在 IM 里口头同步。
workflow 04
AgentThread A2A
实时咨询、脑暴、架构评审走 thread 化 A2A；只用于讨论，不替代任务系统。
workflow 05
Cron Health Check
定时检查 gateway、provider、memory、代理、关键端口。Agent 常驻运行就要有运维意识。
workflow 06
外部动作确认
邮件、发帖、群消息、删除、付款、公开发布前必须确认。让 agent 大胆内部行动、谨慎外部行动。

multi-agent

多 Agent 协作：先定协议，再定角色

Hermes 能接很多入口；我们的教训是：真正落地时，“任务、讨论、通知”必须分层，不能混在一个聊天流里。

task
Issue / Ticket
需要行动、交付、验收：必须进入任务系统。
discussion
A2A / Thread
咨询、脑暴、口径确认：可以实时对话，但要留痕。
human update
Message
进展同步给人类：不是给 bot 派活。
hard lesson
群里看见 ≠ Agent 收到
Telegram/Slack/Discord 的 bot-to-bot、@mention、thread 都有平台限制。一定要用可验证的 agent 通道。

模型治理：不要所有环节都用同一个模型

token-cost-case

成本治理案例：主模型升级后，先查隐性消耗

今天的优化不是“把主模型换便宜”，而是确认 gpt-5.5 只出现在真正需要高质量判断的地方。

symptom
辅助任务走主模型
多个 profile 的 session_search / skills_hub / flush_memories / title_generation 仍是 auto，实际解析到 gpt-5.5。
measurement
先做基线
备份配置、解析 auxiliary client、跑 Gemini proxy smoke test、查 24h session 体积。
fix
分层而非降级
主模型保留 gpt-5.5；压缩、搜索、skill 匹配、标题、memory flush 固定到 Gemini flash-lite。
原则：主模型负责决策，辅助模型负责搬运、压缩、检索和后台判断。

hindsight-perf-case

记忆性能案例：Recall 慢，不等于模型差

Hindsight 之前出现 190s recall / 500 错误。根因是 worker、consolidation 和本地 reranker 资源争用。

before
190s recall
consolidation 占满 8–10 个 worker slot，RSS 约 4.6GB，前台 recall 被拖死。
control
限并发
worker=4；consolidation slot=1；reranker max_concurrent=1；candidate=80。
after
1.5–3s
禁用 entity expansion 的直接 recall 恢复到秒级；默认工具 recall 也恢复可用。
# 关键不是关闭记忆，而是限制后台任务别抢前台检索
HINDSIGHT_API_WORKER_MAX_SLOTS=4
HINDSIGHT_API_WORKER_CONSOLIDATION_MAX_SLOTS=1
HINDSIGHT_API_RERANKER_MAX_CANDIDATES=80

ops-checklist-cost

优化方法论：每次改配置都要闭环验证

今天这轮优化的可复用流程：先备份和测基线，再小步修改，最后用 smoke test 和日志验证。

1

Backup

备份 config、AGENTS、systemd service

2

Resolve

用代码解析实际 provider / model / base_url

3

Smoke

Gemini proxy、agent API、Hindsight health

4

Patch

固定 auxiliary、cron、worker 限流

5

Verify

端口、服务、recall、日志、session 增长

team rule
优化不是省钱脚本，而是可回滚的运维变更
每个变更都要有备份、可观测指标和明确的回滚路径。

ops

运维：Agent 也需要 SRE 思维

Hermes 是常驻服务，就要按服务治理：健康检查、日志、回滚、权限、成本。

health
active ≠ healthy
systemd active 不代表 DB、provider、gateway 都可用。
logs
先看日志
journal、session、request dump、provider 响应要能定位。
smoke test
改后验证
改配置后跑最小请求，不要等用户发现。
secrets
分离管理
不要把 key 写进文档或公开仓库。
proxy
网络显式
CLI、systemd、sandbox 继承的代理环境可能不同。
permissions
外部确认
发消息、发帖、邮件、删除、付款都要确认边界。

anti-patterns

反模式：这些做法会让 Hermes 退化

anti 01
只聊天不行动
没有工具调用、没有验证、没有产物。
anti 02
IM 当任务系统
群聊里派活，最后没有 owner、状态、验收。
anti 03
Memory 当垃圾桶
所有东西都塞常驻记忆，prompt 膨胀且污染判断。
anti 04
配置不验证
改完不跑 smoke test，线上慢慢坏掉。
anti 05
旧 skill 不清理
过期 SOP 会稳定地诱导 agent 走错流程。
anti 06
过早多 Agent
单 agent 流程都没跑顺，多 agent 只是增加噪音。
